# 基于思维链的 BGP 异常溯源分析 —— 已完成工作梳理
## 一、与毕设任务书的对应关系
| 任务书环节 | 项目实现模块 | 说明 |
| --- | --- | --- |
| **观测** | `data_provider`、告警上下文 | 路由事件观测与数据采集 |
| **关联** | `graph_rag`、`topology`、`geo`、`neighbor` | 拓扑与多源数据关联分析 |

## 二、各模块功能与在思维链中的位置
### 1. 观测层：BGP 路由事件的观测与采集
`tools/data_provider.py`

+ 对接 RIPEstat API，获取真实观测数据：
    - **RPKI 验证**：`rpki-validation`（前缀与 Origin 授权）
    - **AS 信息**：`as-overview`（运营商、持有者等）
    - **地理位置**：`maxmind-geo-lite`、`whois`（IP/ASN 归属地）
+ 功能定位：为后续关联、推理和归因提供**观测数据基础**。

**告警上下文 **`alert_context`

+ 包含 `prefix`、`as_path`、`detected_origin`、`expected_origin`，对应任务书中的“观测到的异常路由事件”。

---

### 2. 关联层：拓扑与多源信息的关联分析
`tools/graph_rag.py`**（Neo4j 知识图谱）**

+ 用 Cypher 查询实现：
    - 前缀归属：前缀 → 合法 Origin（ORIGINATES 关系）
    - 拓扑路径：Origin 与 Owner 之间 shortestPath，检测“拓扑瞬移”、伪造连接
+ 功能：把 AS 关系、前缀归属和图结构做**结构化关联**，为推理提供依据。

`tools/topology.py`

+ 实现 Valley-Free 违规检测，识别 Tier-1 → 非 Tier-1 → Tier-1 的**路由泄露模式**。
+ 功能：利用商业拓扑与路由策略做**关系约束关联**。

`tools/geo.py`

+ 比较前缀注册地与 Origin AS 注册地，发现**地理不一致**，辅助判定异常。
+ 功能：引入地理位置作为**额外关联维度**。

`tools/neighbor.py`

+ 解析 AS_PATH 首跳，结合 `BGPDataProvider.get_as_info` 定位传播源。
+ 功能：对异常路由的**传播路径做关联**。

`tools/import_real_word.py`

+ 从 CAIDA 下载并解析 `as-rel.txt`，导入 Neo4j 构建真实 AS 商业关系图。
+ 功能：为关联分析提供**真实拓扑数据**。

---

### 3. 推理层：思维链式结构化推理
`bgp_agent.py`**（核心 Agent）**

+ **Prompt 设计**：要求按 Path Forensics、Route Leak 等方法论分步分析，模拟专家思路。
+ **CoT 实现**：强制输出 `thought_process`，显式展示推理过程。
+ **多轮推理**：最多 3 轮，每轮包含：
    - 思考并决定调用哪个工具
    - 执行工具获得结果
    - 根据结果继续推理
+ **RAG 辅助**：先通过 RAG 检索相似历史案例，再让 LLM 在分析时参考，增强推理合理性。
+ 功能：对应任务书“基于已知网络协议和历史数据，运用推理能力进行逻辑推导”。

`tools/rag_manager.py`

+ 使用 ChromaDB + SentenceTransformer 构建向量库，按 prefix、as_path、origin 做语义检索。
+ 存储字段：`scenario_desc`、`analysis_logic`、`conclusion` 等。
+ 功能：为推理提供**可解释的参考案例**。

---

### 4. 归因层：精确定位问题根源
`tools/bgp_toolkit.py`** 中的 **`path_forensics`

+ 解析 AS_PATH，提取 Observed Origin（路径末端）与 Expected Owner。
+ 判断逻辑：
    - Origin ≠ Expected Owner → Origin 为首要嫌疑人（Primary Suspect）
    - Origin = Expected Owner → 转向 Route Leak，分析上游 AS。
+ 功能：对应任务书“精确定位问题根源”，是**归因的核心工具**。

`tools/authority.py`

+ 调用 RIPEstat 的 `rpki-validation`，验证 Origin 是否有 RPKI 授权。
+ 功能：用 RPKI 做**权威归因验证**，区分合法与非法宣告。

`bgp_agent.py`** 中的 **`final_decision`

+ 输出 `status`（MALICIOUS / LEAK / BENIGN）、`attacker_as`、`summary`。
+ 功能：形成**可解释的结案结论**，支持用户理解异常原因与影响。

---

### 5. 数据生成与评估
`tools/gen_forensics_data.py` / `auto_generator/auto_generator.py`

+ 用 LLM 生成 Direct Hijack、Path Forgery、Route Leak 等合成案例。
+ 每个案例包含 `analysis_logic`，即“如何通过 AS_PATH 锁定攻击者”的思维链。
+ 功能：为 RAG 知识库和 Agent 训练/评估提供**高质量溯源案例**。

`build_vector_db.py`

+ 将 `full_attack_cases.jsonl` 导入 ChromaDB。
+ 功能：搭建溯源专用的 **RAG 知识库**。

`performance_test.py`

+ 从 `data/test_cases.json` 加载案例，调用 Agent 进行批量溯源。
+ 指标：正确识别 attacker AS 的比例（命中率），并给出评级。
+ 功能：对溯源能力和**决策可信度**做量化评估。

---

## 三、整体流程与任务书的一致性
1. **观测**：`alert_context` + RIPEstat 等外部 API 提供路由事件与观测数据。  
2. **关联**：Graph RAG、topology、geo、neighbor 等多工具做拓扑与多源信息关联。  
3. **推理**：Agent + RAG 进行多轮、结构化推理，并显式输出 `thought_process`。  
4. **归因**：`path_forensics`、`authority_check` 结合 `final_decision` 给出 attacker AS 和结论摘要。

## 四、工作要点
1. **层次化 CoT 设计**：通过 System Prompt 和多轮推理循环，显式约束“观测 → 关联 → 推理 → 归因”的步骤。
2. **工具调用式推理**：推理过程依赖 path_forensics、graph_analysis、authority_check 等工具，增强可验证性和可解释性。
3. **知识增强**：RAG 检索历史溯源案例 + Neo4j 拓扑图谱，使推理基于“网络协议 + 历史数据”。
4. **多类型异常支持**：支持 Origin Hijack、Route Leak、Path Forgery 等，与任务书中的“复杂网络异常”要求相符。
5. **可评估性**：`performance_test.py` 提供溯源准确率与评级，支撑“决策可信度”的讨论。
