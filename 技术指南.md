# BGP 异常溯源系统技术指南

本文档是本项目的使用与实现说明，覆盖运行环境、输入输出、核心技术细节、完整操作流程与常见问题。

---

## 1. 项目目标与范围

本系统面向 BGP 异常场景（前缀劫持、路由泄露、路径伪造），构建了一个可解释的溯源分析链路：

- **观测**：抓取真实 BGP updates（RIPE RIS MRT / RIPEstat BGPlay）
- **关联**：融合路径、拓扑、授权等多源信息
- **推理**：LLM 多轮推理 + 工具调用（Tool-Augmented CoT / ReAct）
- **归因**：输出最可能攻击者 AS、状态与置信度

---

## 2. 系统架构

```text
[输入事件配置]
  data/test_events.json
          |
          v
[Step1 数据抓取与过滤]
  scripts/step1_collect_events.py
  -> tools/update_fetcher.py
  -> tools/ris_mrt_fetcher.py / RIPEstat BGPlay
          |
          v
[事件缓存]
  data/events/<event_id>/
  - meta.json
  - suspicious_updates.json
  - raw_bgplay.json(可选)
          |
          v
[Agent 溯源]
  bgp_agent.py
  - RAG 检索: tools/rag_manager.py
  - 工具调用: tools/bgp_toolkit.py
          |
          v
[评测与报告]
  performance_test.py
  report/forensics_*.json
```

## 3. 运行环境（必要条件）

### 3.1 Python 与系统

- Python 3.10+（建议 3.10 或 3.11）
- Linux / macOS / Windows（WSL）均可
- 可访问外网（用于 RIPE 接口与模型 API）

### 3.2 Python 依赖

在项目根目录执行：

```bash
pip install openai chromadb sentence-transformers neo4j requests aiofiles tqdm tabulate mrtparse
```

### 3.3 外部服务

- **LLM API**：默认通过 OpenAI 兼容接口调用（当前代码中在 `bgp_agent.py`）
- **RIPE 数据源**：
  - RIS MRT: `https://data.ris.ripe.net/`
  - RIPEstat BGPlay: `https://stat.ripe.net/data/bgplay/data.json`
- **Neo4j（可选）**：启用 `graph_analysis` 的在线图查询能力

### 3.4 配置项说明

- 知识库配置：`config/knowledge_base.json`
- RAG 持久化目录：`rag_db/`
- 事件输入配置：`data/test_events.json`

> 安全建议：API Key 不应硬编码，建议改为环境变量读取。

## 5. 输入规范（必须掌握）

### 5.1 事件输入：`data/test_events.json`

字段示例：

```json
[
  {
    "prefix": "8.8.8.0/24",
    "victim": "15169",
    "attacker": "4761",
    "start_time": "2014-04-01T08:00:00",
    "end_time": "2014-04-01T18:00:00",
    "source": "anomaly"
  }
]
```

字段说明：

- `prefix`：目标前缀（必填）
- `victim`：合法 Owner AS（必填）
- `attacker`：真值攻击者 AS（评估用；良性可填 `None`）
- `start_time` / `end_time`：时间窗口（ISO8601）
- `source`：事件标签（`anomaly` 或 `benign`）

### 5.2 直接测试输入（可选）

- `data/test_cases.json`：无 Step1 抓取时可直接测试

## 6. 输出规范（你将得到什么）

### 6.1 Step1 输出：`data/events/<event_id>/`

- `meta.json`：事件元数据（来源、时间、可疑条数等）
- `suspicious_updates.json`：过滤后的可疑 updates
- `raw_bgplay.json`：原始抓取数据（可选）

### 6.2 Agent 输出：`report/forensics_*.json`

关键字段：

- `target`：目标事件或批量 updates
- `rag_context`：检索到的参考案例文本
- `chain_of_thought`：每轮推理/工具调用记录
- `final_result`：最终判定（状态、攻击者、置信度、摘要）

## 7. 使用流程（推荐）

### 流程 A：完整闭环（推荐）

#### Step 0. 构建 RAG 案例库

```bash
python auto_generator/auto_generator.py
python build_vector_db.py
```

#### Step 1. 抓取并过滤真实 updates

```bash
python scripts/step1_collect_events.py --input data/test_events.json --source ris_mrt
```

可选参数：

- `--source ris_mrt`（默认，支持历史）
- `--source ripestat`（BGPlay，适合新时间段）
- `--source auto`（RIS 优先，失败回退 BGPlay）

#### Step 2. 批量评测

```bash
python performance_test.py --events
```

### 流程 B：快速验证（无 Step1）

```bash
python bgp_agent.py
python bgp_agent.py batch
python performance_test.py
```

## 8. 核心技术细节

### 8.1 四步法过滤（观测层）

在 `tools/update_fetcher.py` / `tools/ris_mrt_fetcher.py` 中实现：

1. 前缀过滤：仅保留目标前缀相关 updates
2. Origin 校验：`detected_origin != expected_origin` 标记异常
3. 时间窗口约束：仅保留时间区间内事件
4. Valley-Free 检测：识别 Tier1 -> 非Tier1 -> Tier1 异常模式

### 8.2 RAG 检索策略（关联层）

`tools/rag_manager.py` 当前已采用增强策略：

- **结构化过滤 + 向量召回**：先按攻击族（hijack/leak/forgery）过滤候选
- **两阶段检索**：粗召回（默认 top-15）后重排
- **动态 top-k**：候选质量高时只返回 top-1，避免噪声案例干扰
- **阈值拒答**：低置信度时返回“未找到高置信案例”
- **批量签名聚合**：按 `(prefix, detected_origin, expected_origin, path_tail)` 聚合更新并按频次加权

这套策略的目标是降低 `updates` 稀疏时的误检索风险。

### 8.3 CoT + 工具调用（推理层）

`bgp_agent.py` 中最多 3 轮推理，循环如下：

1. LLM 输出 `thought_process` + `tool_request`
2. 调用工具（如 `path_forensics` / `authority_check` / `graph_analysis`）
3. 将工具结果回注给 LLM 继续推理
4. 输出 `final_decision`

并在 Prompt 中约束：**RAG 仅作参考，若与工具证据冲突，以工具证据为准**。

### 8.4 主要工具能力（归因层）

- `path_forensics`：提取 origin 与路径关键跳点，给出候选攻击者
- `authority_check`：RPKI 授权验证
- `graph_analysis`：图谱拓扑关系辅助验证（Neo4j 可选）
- `topology_check` / `geo_check` / `neighbor_check`：补充证据

## 10. 参数与调优建议

### 10.1 数据抓取

- `tools/ris_mrt_fetcher.py` 中 `MAX_FILES` 控制单次窗口抓取上限
- 如果事件窗口很长，建议分段抓取或增大该值

### 10.2 RAG 检索

- `tools/rag_manager.py`：
  - `recall_k`：粗召回数量（默认 15）
  - `reject_distance`：低置信拒答阈值（默认 0.75）

建议在验证集上调参，观察准确率与稳定性变化。

## 11. 常见问题与排查

### 11.1 报错 `ModuleNotFoundError`

缺依赖，重新执行：

```bash
pip install openai chromadb sentence-transformers neo4j requests aiofiles tqdm tabulate mrtparse
```

### 11.2 RAG 检索质量不稳定

- 先重建向量库：`python build_vector_db.py`
- 检查案例数据质量与类别分布
- 调整 `recall_k` 与 `reject_distance`

### 11.3 RIS 无数据

- 检查时间窗口是否合理
- 切换 `--source auto` 或 `--source ripestat`
- 对历史事件可适当扩大抓取窗口

### 11.4 Neo4j 不可用

不影响主流程，可先使用离线模式；启用图谱后再增强拓扑证据。

## 12. 推荐最小复现实验

1. 配置 `data/test_events.json`（至少 3 个异常 + 1 个良性）
2. 执行 Step1 抓取过滤
3. 执行 Step2 评测
4. 检查 `report/` 下输出：
   - 是否含完整 `chain_of_thought`
   - `final_result` 是否给出 attacker 与置信度
